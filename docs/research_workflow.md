# Research Workflow

This repository supports experiments on container live migration with CRIU and
runc. The focus is the migration behavior of a Flask workload under pre-copy and
post-copy migration, measured from a client/monitor perspective.

`clm` is the experiment runner used to orchestrate runs, collect measurements,
and build analysis outputs. It is not the primary research object.

## Research Questions

- How do pre-copy and post-copy differ in client-visible availability?
- Which migration phases dominate downtime?
- How long is the post-copy source application frozen from CRIU freeze/seize to
  CRIU unfreeze?
- How much downtime is caused by CRIU restore/checkpoint work, application
  readiness, storage transfer, and VIP cutover behavior?
- How do idle, CPU, transfer, stream, and HTTP load profiles affect the result?

## Host Roles

| Role | Purpose |
|---|---|
| Source | Runs the workload container before migration and creates CRIU checkpoints. |
| Destination | Restores the runc container and takes over the VIP. |
| Monitor | Generates client-side probes, optional load, and analysis artifacts. |

The lab used a source, destination, and monitoring host on one L2 network. The
example configuration uses host networking, a shared `/mnt/criu` path, and a VIP
on port `8080`. Concrete IP addresses are testbed-specific and should be
replaced in independent reproductions.

## Experimental Object

- Container runtime: `runc`
- Checkpoint/restore: CRIU
- Workload: Flask/Gunicorn container
- Client access path: VIP-based HTTP service
- Migration methods:
  - `precopy`
  - `postcopy` with CRIU lazy pages

## Current `clm` Defaults

These defaults describe the current experiment runner behavior, not every
standalone script default.

| Area | Current setting |
|---|---|
| Workload process | Gunicorn, 1 worker, 4 threads. |
| Pre-copy | `pre_dump_rounds: 0`, `image_mode: shared`, `tcp_established: 1`. |
| Post-copy storage | Source images on shared storage; destination restore from a local image copy and local bundle. |
| Post-copy forwarding | Enabled, `iptables_dnat`; prepared before checkpoint and activated after the first direct destination HTTP 200. |
| Post-copy readiness | Destination `/health`, 3 stable successes, 200 ms interval, 10 s timeout. |
| Post-copy warmup | None in the standard v22 script. Legacy configuration fields may still be recorded as metadata. |
| VIP cutover | Make-before-break: destination VIP add/verify and GARP before source VIP removal. |
| Monitoring | VIP/src/dst HTTP and L4 at 50 ms; stream/info/counter enabled by default. |

## Run Structure

Each run should persist enough information to re-analyze the result:

| Path | Contents |
|---|---|
| `meta/` | Git commit, config snapshot, host and tool versions, clock offsets. |
| `events/` | Migration events in NDJSON format. |
| `monitor/` | HTTP, L4, stream, download, upload, info, and counter logs. |
| `migrate/` | Migration stdout/stderr and CRIU logs where available. |
| `summary.json` | Per-run metrics generated by the analyzer. |
| `status.json` | Run state, error reason, and exit codes. |

Batch analysis writes aggregate outputs such as `metrics.csv`,
`summary_stats.csv`, `summary_stats.json`, `downtime_segments.csv`, and plots.

## Measurement Perspective

Client-visible VIP reachability from the monitor host is the primary service
availability metric. Post-copy also has a separate primary internal metric: the
source application freeze from the first CRIU freeze/seize event through CRIU
unfreeze. Neither metric replaces the other.

| Signal | Meaning |
|---|---|
| Source CRIU freeze | Exact interval in which the source application cannot execute. |
| Checkpoint event window | Script-level upper bound around the source checkpoint process. |
| VIP HTTP | End-to-end application availability from the client perspective. |
| VIP L4 | TCP reachability of the service port through the VIP. |
| Direct `src`/`dst` | Infrastructure handoff behavior without the VIP path. |
| Stream/transfer continuity | Disconnects, gaps, and throughput during migration. |

VIP metrics and direct `src`/`dst` metrics intentionally answer different
questions. In post-copy with temporary source-to-destination forwarding, direct
handoff metrics can remain high while VIP downtime is much smaller.

The current post-copy design and its v1-v23 forensic history are documented in
[Post-Copy Freeze-Path Forensics](archive/postcopy_freeze_forensics_2026-07-16.md).

## Scenario Matrix

The paper campaign uses the following load scenarios:

| Scenario | Description |
|---|---|
| `idle` | No explicit load process beyond monitoring. |
| `cpu` | CPU-oriented requests to `/heavy`. |
| `download` | Egress transfer through `/download`. |
| `upload` | Ingress transfer through `/upload`. |
| `stream` | Long-lived stream through `/stream`. |
| `wrk1`, `wrk2`, `wrk3` | HTTP load profiles generated with `wrk`. |

The scenario workbook [scenario_workbook.xlsx](scenario_workbook.xlsx) contains a
compact tabular summary of scenario parameters and run flow.

## Reproducibility Requirements

- Record `config/env.yaml` and analysis configuration with every run.
- Record runc, CRIU, kernel, OS, and Docker versions.
- Use a stable shared artifact path or document an equivalent copy strategy.
- Use the monitor host timestamps as the primary client-side time base.
- Record clock offset estimates for event correlation.
- Keep failed or excluded runs visible in analysis outputs with an exclusion
  reason.
- Report robust statistics such as median, P95, IQR, and explicit sample counts.
